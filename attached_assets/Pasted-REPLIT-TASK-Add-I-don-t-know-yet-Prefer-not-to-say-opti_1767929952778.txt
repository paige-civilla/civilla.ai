REPLIT TASK: Add “I don’t know yet” + “Prefer not to say” options to ALL onboarding fields + make them editable later

GOAL
1) Every onboarding field should allow:
   - “I don’t know yet” (unknown)
   - “Prefer not to say” (prefer_not_to_say)
2) Store these values safely without breaking validation.
3) Users can update everything later in:
   - Account Settings (user-level fields)
   - Case Settings (case-level fields)

----------------------------------------
A) STANDARDIZE VALUES (DO THIS FIRST)
----------------------------------------
Use these exact string constants everywhere:

unknown
prefer_not_to_say

Do not use null for these choices unless field is optional already. These two strings make it explicit and easier to filter later.

  [I don’t know yet] [Prefer not to say]

Component API:

----------------------------------------
D) BACKEND / SCHEMA: ACCEPT THESE VALUES
----------------------------------------
We must ensure saving onboarding does not fail validation.

1) shared/schema.ts
For any Zod schema fields used in onboarding payloads:
- Change validation to allow:
  z.string().min(1) OR literal("unknown") OR literal("prefer_not_to_say")
Example pattern:
z.union([z.string().min(1), z.literal("unknown"), z.literal("prefer_not_to_say")])

2) For fields that are currently typed as Date or number:
Option A (recommended for speed):
- Store them as TEXT in DB (or keep existing column if already text)
- Validate in frontend when real value is provided
- If already stored as date/number, do this:
  - Keep DB column as-is
  - Add parallel “*_raw” text field for onboarding only (more work) OR
  - Treat unknown/prefer_not_to_say as NULL and keep a separate “*_status” column (more work)

Fastest safe approach:
- On save, if value is "unknown" or "prefer_not_to_say", send null for date/number fields AND also store a companion status field.

So:
D2) If any onboarding fields are date/number in DB today:
- Add companion columns:
  - <field>_status TEXT nullable
  values: null | "unknown" | "prefer_not_to_say"

Example:
hearing_date DATE
hearing_date_status TEXT

Then:
- If user picks unknown/prefer_not_to_say:
  hearing_date = null
  hearing_date_status = "unknown" | "prefer_not_to_say"
- If user picks real date:
  hearing_date = actual date
  hearing_date_status = null

3) server/db.ts
- Add ensure migrations for any *_status columns added.

4) server/routes.ts
- When saving onboarding payloads, map sentinel strings to:
  - string fields: store sentinel directly
  - date/number fields: store null + status flag
